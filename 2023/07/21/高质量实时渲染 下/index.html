基于物理的渲染（PBR）

  • 渲染中的一切都应基于物理
  • 材质、照明、相机、光传输等。
  • 不仅仅是材质，但通常被称为材质

RTR中的PBR材质

  • RTR远远落后于离线渲染
  • RTR中的 “PB “通常不是基于物理的
  • 对于表面，主要是微表面模型（使用不当，因此不是PBR）和迪斯尼原理BRDF（艺术家友好，但仍然不是PBR）
  • 对于体，主要侧重于快速和近似的单散射和多散射（云、头发、皮肤等）
  • 通常没有太多的新理论，但有很多实现hacks*。
  • 性能（速度）仍然是需要考虑的关键因素

基本概念

  • 2018 年，英伟达发布了 GeForce RTX 系列（图灵架构）
  • RTX 到底能做什么？每秒 10 Giga 射线 == 每个像素 1 个样本（实时应用）

1 SPP path tracing =

1 rasterization (primary) + 摄像机到观察点的射线，相当于光栅化

1 ray (primary visibility) + 观察点到光源的线，观察是否被遮挡

1 ray (secondary bounce) + 二次弹射打到点的线

1 ray (secondary vis.) 二次弹射被打到点的线是否和光源间无遮挡

1 SPP = 噪音极高的结果

关键技术：去噪

目标（1 SPP）

  • 质量（无过度模糊、无人工痕迹、保留所有细节……）。

不可能完成的方法

  • 剪切滤波系列（SF、AAF、FSF、MAAF…）
  • 其他离线滤波方法（IPP、BM3D、APR…）

  • 也会模糊边界
  • 但边界是我们想要保留的高频率

观察

  • 边界 <-> 急剧变化的颜色

想法

  • 如果像素 的颜色与其他像素相差太大，则减少像素 的贡献。
  • 只需为内核添加更多控制

双侧联合滤波

观察

  • 高斯滤波：1 个指标（距离）
  • 双边过滤：2 个指标（位置距离和颜色距离）
  • 交叉/联合双边滤波使用更多 “特征 “来更好地指导滤波

特别适用于路径追踪渲染结果的去噪！

渲染方面的独特优势

  • 大量被称为 G-buffers 的free特征
  • 法线、深度、位置、对象 ID 等，大部分是几何图形
  • G-buffers无噪声，因为它们与多次弹射无关，效果更好

Notes

  • 指标本身无需标准化，滤波过程会进行归一化处理
  • 高斯函数不是唯一的选择，任何随 “距离 “递减的函数都适用，指数（绝对值）、余弦（clamped）等

关于抗锯齿的说明

  • 附加说明 1

    • MSAA（Multisample） VS SSAA（Supersampling）

    • MSAA：提高性能。同一基元只着色一次。跨像素重复使用样本

Deferred Shading

最初是为了节省着色时间而发明的

  • 考虑光栅化过程
  • Triangles -> fragments -> depth test -> shade -> pixel
  • 每个片段都需要着色（在什么情况下？）
  • 复杂度： O(#fragment * #light)

主要观察结果

  • 大多数片段不会出现在最终图像中
  • 由于深度测试/遮挡
  • 能否只对可见的片段进行阴影处理？

修改光栅化流程

  • 只需对场景进行两次光栅化
  • 第 1 次：不着色，只更新深度缓冲区
  • 第 2 次：栅格化过程相同（为什么这能保证只对可见片段着色？）
  • 隐含的假设是，对场景进行光栅化比对所有未见片段进行着色要快得多（通常情况下确实如此）
  • 复杂度： O(#fragment * #light) -> O(#vis. frag. * #light)

问题

  • 难以进行抗锯齿处理
  • 但 TAA 几乎完全解决了这一问题

Tiled Shading

改进：tiled shading

  • 将屏幕细分为例如 32x32 的tile，然后为每个tile着色

主要意见

  • 并非所有lights都能照亮特定的tile
  • 主要原因是square随距离衰减（！）
  • 复杂性： O(#vis. frag. * #light) -> O(#vis. frag. * avg #light per tile)

Clustered Shading

进一步改进：Clustered Shading

  • 将每个色块进一步细分为不同的深度段
  • 本质上是将视图圆锥体细分为三维网格

主要观察结果

  • 每个瓦片的深度范围可能相当大
  • 因此，可能会有很多灯光被识别为有可能照亮平铺区域
  • 但有些灯光可能只能照亮很小的深度范围
  • 复杂性： O(#vis. frag. * avg #light per tile) -> O(#vis. frag. * avg #light per cluster)

Level of Detail Solutions

Level of Detail (LoD) 非常重要

  • 回顾：texture MIPMAP-ing
  • 选择正确的level of detail可以节省计算量

使用multiple levels of detail

  • RTR 行业通常称之为 “级联(cascaded)”

实例

  • 级联阴影贴图
  • 级联 LPV

主要挑战

  • 不同层次之间的过渡
  • 通常需要在边界附近进行一些重叠和混合

另一个例子：geometric LoD

  • 回顾：预先生成一组具有不同 #tri. 的简单 obj.
  • 根据与摄像机的距离，选择合适的对象来显示（或对象的一部分，但任何三角形都不会大于一个像素）
  • Popping artifacts？交给 TAA 吧！
  • 这是 UE5 中的 Nanite（当然，Nanite的功能远不止这些）

仅供参考，存在一些（强烈的）技术困难

  • 不同的地方有不同的级别，接缝如何处理？
  • 动态加载和调度不同级别，如何充分利用缓存和带宽等？
  • 使用三角形或几何纹理表示几何图形？
  • 剪切和剔除以提高性能？
  • …

Global Illumination解决方案

从本课程中，我们可以看到

  • 回想一下，屏幕空间光线追踪 (SSR) 什么时候会失败？
  • 除了 RTRT 之外，没有一种全局照明解决方案能完美地应对所有情况。
  • 但完全使用 RTRT 在当前一代产品中仍然成本过高
  • 因此，业界倾向于使用混合解决方案

例如，GI 的可能解决方案包括

  • SSR 用于粗略近似 GI
  • SSR 失效后，切换到更复杂的光线追踪
  • 硬件（RTRT）或软件（？）

软件光线追踪

  • HQ SDF 用于近距离的单个物体
  • 整个场景的 LQ SDF
  • 如果有强定向光/点光源，则使用 RSM
  • 在 3D 网格中存储辐照度的探测器（动态漫反射 GI 或 DDGI）

硬件光线追踪

  • 不一定要使用原始几何体，但可以使用低多边形代理
  • 探测器（RTXGI）

将突出显示的解决方案混合使用，可在 UE5 中获得Lumen效果
